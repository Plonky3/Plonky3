Abstract
Proof-carrying data (PCD) is a powerful cryptographic primitive for computational integrity
in a distributed setting. State-of-the-art constructions of PCD are based on accumulation
schemes (and, closely related, folding schemes).
We present WARP, the first accumulation scheme with linear prover time and logarithmic
verifier time. Our scheme is hash-based (secure in the random oracle model), plausibly postquantum secure, and supports unbounded accumulation depth.
We achieve our result by constructing an interactive oracle reduction of proximity that works
with any linear code over a sufficiently large field. We take a novel approach by constructing
a straightline extractor that relies on erasure correction, rather than error-tolerant decoding
like prior extractors. Along the way, we introduce a variant of straightline round-by-round
knowledge soundness that is compatible with our extraction strategy.
Keywords: interactive oracle reductions; accumulation schemes; linear codes

Introduction

Proof-carrying data (PCD) [CT10] is a powerful cryptographic primitive that allows mutually
distrustful parties to perform distributed computation in an efficiently verifiable manner. PCD
generalizes incrementally verifiable computation (IVC) [Val08], and has applications to enforcing
language semantics [CTV13], verifiable MapReduce computations [CTV15], image/video provenance [NT16; ZYORJ24], memory-efficient succinct arguments [NDCTB24], consensus protocols
[BCG24], blockchains [Min; KB20; BMRS20; CCDW20], and more.
PCD was initially achieved via recursive proof composition of succinct arguments [BCCT13;
BCTV14]. Subsequently, a successful line of work shows how to achieve highly-efficient PCD from
a more lightweight primitive, accumulation schemes (and folding schemes) [BGH19; BCMS20;
BCLMS21; KST22; BC23; KS24; EG23; BMNW25a; BC24; FKNP24; BMNW25b; BC25; NS25].
In this paper we present the first accumulation scheme with linear prover time and logarithmic
verifier complexity. We review accumulation schemes below, and refer the reader to the cited papers
for how PCD is obtained from them.
Accumulation schemes. Informally, an accumulation scheme ACC = (PACC , VACC , DACC ) for an
NP relation R enables verifiably propagating the bit that equals the conjunction of any number of
membership checks for R, without holding onto the verified instance-witness pairs. This is done by
maintaining an accumulator acc. For each instance-witness pair to be accumulated, the prover PACC
updates acc and outputs a short proof pf attesting to the update’s correctness, which the verifier
VACC checks. Afterwards, the accumulator acc is given to the decider DACC , which outputs a bit.
Provided all checks by VACC succeeded, DACC ’s output equals the conjunction of the membership
checks on the accumulated instance-witness pairs.
In fact, we consider a refined notion known as a split accumulation scheme [BCLMS21], in which
the accumulator acc is divided into a short “instance part” acc.x and long “witness part” acc.w.
The verifier VACC receives only instance parts, whereas the prover PACC and decider DACC receive
both instance and witness parts. This refined notion enables particularly efficient accumulation,
since the verifier VACC can be succinct in the witness parts.
More formally, an accumulation scheme ACC = (PACC , VACC , DACC ) for R works as follows.
• Completeness. Let (x, w) ∈ R and acci be such that DACC (acci ) = 1. Then PACC (acci , x, w)
outputs (acci+1 , pf) such that
VACC (acci .x, x, acci+1 .x, pf) = 1 and DACC (acci+1 ) = 1 .
• Knowledge soundness. There exists an efficient deterministic extractor EACC such that, for every
acci .x, x, acci+1 , pf that are efficiently generated, if
VACC (acci .x, x, acci+1 .x, pf) = 1 and DACC (acci+1 ) = 1 ,
then, except for a small error probability, the extractor EACC (given suitable inputs) outputs
witnesses (w, acci .w) such that (x, w) ∈ R and DACC (acci ) = 1 (for acci := (acci .x, acci .w)).
More generally, an accumulation scheme supports any number ℓ1 of instance-witness pairs and any
number ℓ2 of input accumulators; in the above informal description we focus on ℓ1 = ℓ2 = 1.
Prior art: superlinear provers. In PCD schemes obtained from accumulation schemes, the time
complexity of the PCD prover depends linearly on the time complexities of the accumulation prover;
3

moreover, each PCD step recursively proves that the accumulation verifier accepts. While prior
work has constructed accumulation schemes with asymptotically optimal accumulation verifiers, the
same is not true for the prover side. Specifically, in prior constructions of accumulation schemes,
provers run in superlinear time. Below we summarize the efficiency features of prior works, and
postpone a more detailed discussion to Section 1.2.
Prior accumulation schemes. Most accumulation schemes rely on additive homomorphisms
[BCMS20; BCLMS21; KST22; BC23; KS24; EG23] that arise from commitments over prime-order
groups or bilinear groups [Ped92; KZG10]. These constructions require multiscalar multiplications
that involve a superlinear amount of group operations (even with Pippenger’s algorithm [Pip80]).
Some accumulation schemes are based on lattices [BC24; FKNP24; BC25; NS25]. While latticebased linear-time commitments are known [AHIKV17], challenges arise when handling noise growth
towards accumulation. In particular, no prior lattice constructions achieve linear-time provers.
Recent work constructs accumulation schemes in the random oracle model (ROM) by using, in
particular, Merkle commitment schemes [BMNW25b]. Their prover runs in quasilinear time, due
to the use of Reed–Solomon codes and polynomial quotients.
Separately, we note that any succinct non-interactive argument directly implies a corresponding
accumulation scheme (where the accumulation decider always accepts). Qualitatively, this implication yields accumulation schemes that are less efficient than direct constructions of accumulation
schemes, since succinct arguments rely on “heavier” tools (to achieve succinctness). Quantitatively,
the efficiency tradeoffs are more nuanced, as explained in Appendix A.

1.1

Our results

We present WARP, an accumulation scheme in the ROM where the accumulation prover runs
in linear time (performs a linear number of field operations and random oracle queries) and the
accumulation verifier runs in logarithmic time (performs a logarithmic number of field operations
and random oracle queries).
We state our main theorem for polynomial equation satisfaction (PESAT), an NP-complete
problem that naturally captures R1CS [BCGTV13], GR1CS [DMS24], CCS [STW23], and more.
Definition 1 (informal). A polynomial equation system over a field F is a tuple (p̂, M, N, k)
specifying a list of N -variate polynomials p̂ = (p̂i )i∈[M ] over F of constant (total) degree. An
instance-witness pair z := (x ∈ FN −k , w ∈ Fk ) is a solution if p̂i (z) = 0 for every i ∈ [M ]. We
write |p̂| for the number of field operations required for evaluating the equation system. (Without
loss of generality, k = O(|p̂|).) The indexed PESAT relation R⊚ (F) is the set of triples (i, x, w)
where i = (p̂, M, N, k), x = x, and w = w such that (x, w) is a solution to i.
Theorem 1 (informal). In the random oracle model (ROM) with security parameter λro , there is
an accumulation scheme ACC = (PACC , VACC , DACC ) for the PESAT relation R⊚ (F) with the following
properties for accumulating ℓ instance-witness pairs and accumulators.
• PACC performs O(ℓ·|p̂|+λ·log k) field operations and O(k) random oracle queries of size ℓ·log |F|
and λro .
• VACC performs O(ℓ · (log N + log M + λ) + λ · log k) field operations, O(ℓ · λ · log k) random oracle
queries of size λro and O(ℓ · λ) random oracle queries of size log |F|.
• DACC performs O(|p̂|) field operations and O(k) random oracle queries of size log |F| and λro .
The protocol has knowledge soundness error ≤ 2−λ , for sufficiently large |F| and λro .

IOR over (almost) every code. To achieve Theorem 1 we follow the recipe of [BMNW25b],
which compiles (suitable) interactive oracle reductions (IORs) into a corresponding accumulation
scheme. Recall that an IOR is an interactive protocol between a prover and a verifier, in which the
prover sends possibly long proof strings that the verifier does not read in their entirety, but rather
queries as an oracle; after the interaction, the verifier queries the proof strings, and either rejects
or outputs a new instance of a prescribed relation. In an IOR, instances themselves might contain
oracle messages, which the verifier can also query. For our purposes, the efficiency metrics that we
aim to minimize are prover time, verifier time, and query complexity.
Our main technical contribution is two IORs that work over any given linear code, inheriting
the efficiency features of the code (up to constants). Theorem 1 follows by fixing a linear code with
linear encoding time and constant relative distance (e.g., the expander code in [GLSTW23] or the
repeat-accumulate-accumulate code in [BCFRRZ25]), and applying the aforementioned recipe.

Almost any code. Theorem 2 requires no structure from the underlying linear code C, apart
from being defined over a sufficiently large field; our IORs do not rely on interleaving or tensoring
of C, contrary to prior linear-time IOPs. The property of mutual correlated agreement [ACFY25]
mentioned in Theorem 2 is known for every linear code C [AHIV17; ACFY25; GKL24; Zei24];
informally, this property states that, except for a small error, if the random linear combination of
ℓ functions has a large agreement set with the code C then this set coincides with the agreement
set of these ℓ functions with the interleaved code C ℓ (see Section 3.6 for a formal definition).
Out-of-domain samples for every linear code. Out-of-domain samples in the Reed–Solomon
proximity testing literature [BGKS20; ACFY24; ACFY25] are widely used to achieve efficient

proximity testing in the list decoding regime. [BGKS20] suggested an abstract definition of outof-domain samples for arbitrary linear codes, but did not use it beyond Reed–Solomon codes. Our
results crucially rely on out-of-domain samples obtained via a carefully chosen specialization of

Straightline extraction without efficient decoding. In the probabilistic proof literature,
achieving straightline knowledge soundness is relatively straightforward when the underlying code
has an efficient (error-tolerant) decoding algorithm (e.g., this is true for Reed–Solomon codes).
Roughly, the extractor runs the decoding algorithm on a function that, by the security analysis, is
sufficiently close to an encoding of a valid witness. When the code is not known to have an efficient
decoding algorithm (this is the case for certain linear-time encodable codes), showing knowledge
soundness is significantly more involved, and usually involves rewinding extraction.
We introduce a weakening of round-by-round knowledge soundness, with straightline extraction, that we show our IORs satisfy regardless of whether the given code has an efficient decoding
algorithm. Moreover, we show that our notion implies (straightline) state-restoration knowledge
soundness, thus sufficing for compilation into a (non-interactive) protocol in the ROM according to
the recipe in [BMNW25b]. Our strategy is not specific to IORs, and we believe that the definitions
and tools in this paper can be applied to obtain straightline extraction for existing IOPs that use
linear-time encodable codes (e.g., [BCGGHJ17; BCG20; BCL22; GLSTW23; ZCF24; BCFRRZ25]).
Small fields. The field size requirements in Theorem 1 and Theorem 2 (which imply that |F| ≥ 2λ )
can be relaxed, enabling the use of smaller fields. Given a field B and linear code C : Bk → Bn ,
WARP can be adapted to accumulate instances of R⊚ (B). We describe this adaption in Appendix D,
which involves sampling challenges from a sufficiently large field extension F of the field B (the
requirements on |F| are as before). Note that WARP is not necessarily linear time with this adaption,
since the accumulation prover would perform a linear number of field operations in the extension
field F. In the (common) setting where |B| = O(1) and |F| ≥ 2λ , a field operation in F induces O(λ)
field operations in B, and thus the prover cost is superlinear. Nevertheless, using extension fields
may be concretely efficient and well-suited for many relations of interest. Achieving an accumulation
scheme that, for every field B, performs a linear number of B-operations remains an open question.

1.2

Related work

The most relevant prior work is hash-based accumulation [BMNW25a; BMNW25b], which we
discuss in Sections 2.1 and 2.2. Below we summarize other prior work on accumulation schemes.
Group-based accumulation. Group-based accumulation schemes (e.g., [BCMS20; BCLMS21;
KST22; KS24]) achieve asymptotically optimal verifier complexity (O(1) group operations). These
constructions rely on a Pedersen commitment [Ped92] or a KZG commitment [KZG10] of the
λ·n
witness, which involves a large multi-scalar-multiplication. This takes O( log(λ·n)
) group operations
via Pippenger’s algorithm [Pip80], making them unsuitable for linear-time accumulation.
Lattice-based accumulation. Lattice-based accumulation schemes can use [Ajt96] commitments, which are partially homomorphic. Nevertheless, the challenge of lattice-based accumulation
lies in controlling the norm growth of the committed (and extracted) witness (thereby enabling unbounded accumulation). A recent exciting line of work [BC24; FKNP24; BC25; NS25] constructs
accumulation schemes overcoming this challenge, but, to our knowledge, no construction achieves
a linear-time prover (either because of superlinear commitment costs or superlinearity in the accumulation prover). Achieving a lattice-based accumulation scheme that controls norm growth while
6

maintaining a linear-time prover remains an open problem.
Remark 1.1 (concurrent work). [BMMS25] gives a linear-time accumulation scheme (in the ROM)
that matches the verifier complexity of WARP. A few differences limit its concrete efficiency.
• The accumulation relation in [BMMS25] is less expressive than ours, which incurs concrete
overheads. Specifically, their analogue of Construction 5.10 must invoke the sumcheck protocol.
In practice, this requires the prover and verifier to perform more field operations.
• In their analogue of Construction 9.4, the prover sends multiple oracles (at least three, compared
to one in our case). This requires the prover and verifier to make more random oracle queries.
• [BMMS25] only supports proximity bounds within the unique decoding regime. In practice, this
requires the verifier to make more random oracle queries.
Finally, [BMMS25] does not achieve straightline knowledge soundness when the underlying code
is not known to have an efficient error-tolerant decoder. We believe that our techniques can be
applied to their protocols to obtain straightline extraction.

Technical overview

We discuss the main ideas behind our results.

2.1

On hash-based accumulation

We review the general framework that underlies hash-based accumulation schemes.
SNARGs in the ROM. Succinct non-interactive arguments in the ROM are obtained via the
BCS transformation [BCS16]. The starting point is an interactive oracle proof (IOP) [BCS16;
RRR16], which is an interactive protocol between a prover and a verifier in which the prover sends
possibly long messages that the verifier can query as oracles. The BCS transformation relies on:
(i) Merkle commitment schemes (in the ROM) to enable the argument prover to succinctly commit
and locally decommit to the prover’s long messages; and (ii) the Fiat–Shamir transformation (in the
ROM) to enable the argument prover to derive the IOP verifier’s (random) messages. In practice,
by heuristically instantiating the random oracle, this leads to so-called hash-based SNARGs.
Accumulation in the ROM. Accumulation schemes in the ROM are obtained in a way that
mirrors the above strategy [BMNW25a; BMNW25b]; in practice, this leads to hash-based accumulation schemes. Such constructions share many benefits with hash-based SNARGs: a transparent
setup (the choice of hash function to instantiate the random oracle), fast proving (due to flexibility
in field choice and size), plausible post-quantum security,1 and more.
The starting point is relaxed from an IOP to an interactive oracle reduction (IOR) [BCGGRS19;
BMNW25b]. Informally, an IOR is an interactive oracle protocol in which the verifier, rather than
outputting a decision bit, either rejects or outputs an instance for the target relation; instances
may contain oracle messages, which the verifier can also query. See Section 3.2 for a definition.
The BCS transformation can then be adapted to map an IOR into a corresponding noninteractive reduction in the ROM [BMNW25a; BMNW25b]. As discussed in Section 1, we seek
an accumulation scheme ACC = (PACC , VACC , DACC ) for R in the ROM. This directly follows from
two IORs that are the main target in this paper.
• An IOR that reduces checking membership in R to checking that a sent oracle (typically, an
alleged encoding of a valid witness) is close to a certain code-based accumulation relation RC .
• An IOR that reduces ℓ proximity claims to RC to a single proximity claim to RC .
These two IORs are akin to the two main components of many IOPs: an “IOP preamble” that
reduces deciding the relation to a proximity claim, and an IOP of proximity for that claim.
Linear-time proving. A line of work [BCGGHJ17; BCG20; BCL22; GLSTW23] constructs IOPs
with a linear-time prover. These directly lead to linear-time succinct arguments in the ROM (or
in the plain model from linear-time collision resistant hash-function [AHIKV17]). From a technical
perspective, we seek to replicate this success, by designing suitable IORs with a linear-time prover.

2.2

Limitations of prior work

We review the main features and limitations of prior IORs.
1

IORs for every linear code, but with distance loss. [BMNW25a] constructs IORs for
batching proximity claims to a general linear code (via random linear combinations and spotchecking). However the IORs do not preserve distance: the output proximity claim is always for
a (strictly) smaller distance than for the input proximity claims. This limitation translates (after
hash-based compilation) into achieving a bounded-depth accumulation scheme (a weaker notion of
accumulation). Moreover, the IORs are proven secure only in the unique-decoding regime and the
IOR prover sends two oracle messages (all other messages are short); this hinders performance.2
Distance-preserving IORs, but only for Reed–Solomon codes. [BMNW25b] replaces the
core component of [BMNW25a] with a protocol inspired by [ACFY24], using quotients to enforce
in-domain and out-of-domain constraints. The resulting IOR is distance-preserving and is proved
secure in the list-decoding regime, improving the number of queries performed by the verifier.
However, quotienting has drawbacks. First, the protocol is restricted to Reed–Solomon codes,
which have quasilinear encoding time. Second, polynomial quotients require the prover to run an
additional quasilinear-time FFT, and the verifier performs O(ℓ · λ2 ) field operations. Finally, the
IOR prover sends two oracle messages (all other messages are short), which hinders performance.
This paper. We design IORs for proximity claims to a general linear code, enforcing in-domain
and out-of-domain constraints in a new way that avoids quotienting altogether. We address the
other shortcomings as well: our IORs are secure in the list-decoding regime, and the IOR prover
sends a single oracle message (all other messages are short).

2.3

Out-of-domain (and in-domain) sampling for every linear code

We review out-of-domain sampling [BGKS20] and in-domain sampling [ACFY24; ACFY25], techniques that are useful towards constructing IOPPs for Reed–Solomon codes with good soundness
in the list-decoding regime. Then we extend these notions to any linear code C : Fk → Fn .
Throughout, we fix a function f : [n] → F, and let Λ(C, f, δ) be the set of codewords in C at
relative distance ≤ δ from f ; Λ(C, δ) denotes the maximum of Λ(C, f, δ) across all functions f .
Out-of-domain (and in-domain) sampling for RS codes. Consider the special case of a
Reed–Solomon code: C consists of all evaluations on [n] ⊆ F of univariate polynomials over F of
degree less than d (interpret [n] as a domain L in F).
Out-of-domain sampling is a technique that forces the prover to “choose” a single codeword in
Λ(C, f, δ) (if δ ≥ δ(C)/2 then it can be that |Λ(C, f, δ)| > 1). The verifier sends a random element
τ ∈ F (the out-of-domain sample) and the prover answers with an element σ ∈ F; in the honest
case f ∈ C and σ := fˆ(τ ) where fˆ is the univariate polynomial whose evaluation on [n] equals f .
The verifier then considers the claim “there exists u ∈ Λ(C, f, δ) such that û(τ ) = σ”. This forces
the prover to “choose” (at most) one codeword in Λ(C, f, δ) because, with high probability over τ ,
no two codewords u1 , u2 ∈ Λ(C, f, δ) are such that û1 (τ ) = û2 (τ ).
In-domain sampling refers to a claim of the form “f (τ ) = σ” for a random element τ ∈ [n]
(the in-domain sample) and a given response σ ∈ F. Such claims are used to enforce consistency
between different oracles sent by the prover (e.g., see domain shifting in [ACFY24; ACFY25]).
Moreover, prior protocols exploit the fact that, for Reed–Solomon codes, in-domain sampling
can be viewed as a special case of out-of-domain sampling [ACFY24; ACFY25]. In the honest

What happens in general? For a general linear code C, an in-domain claim is a statement of

Codeword batching, with constraints

We outline a (simplified) protocol for codeword batching, a key subroutine in our IOR. The prover
and verifier have access to two (or more) oracles, along with claims about a linear combination of
said oracles. The prover and verifier interact and, at the end of the interaction, the verifier outputs
a list of claims about a single oracle. This is essential for constructing a many-to-one reduction
whose new instance size is independent of the number of old instances.
Definition 3 (informal). A codeword batching protocol with proximity radius δ ⋆ is an interactive (oracle) protocol (P, V) with the following properties.
• Syntax: the verifier V gets as input randomness γ1 , γ2 ∈ F, multilinear evaluation claim (α, µ),
and oracle access to f1 , f2 : [n] → F. The prover P gets as (explicit) input (γ1 , γ2 , α, µ, f1 , f2 ).
At the end of the interaction, V outputs a list of multilinear evaluation claims (τi , σi )i and
(implicitly) an oracle g : [n] → F.
• Completeness: if γ1 · f1 + γ2 · f2 ∈ C[(α, µ)] then g ∈ C[(τi , σi )i ].
• Soundness: for every δ ∈ (0, δ ⋆ ), if ∆(γ1 · f1 + γ2 · f2 , C[(α, µ)]) > δ then, except for a small
error probability, ∆(g, C[(τi , σi )i ]) > δ.
We describe a codeword batching protocol with proximity radius δ ⋆ = 1 (i.e., distance is preserved for every δ ∈ (0, 1)) that has a linear-time prover and small soundness error.

2.7

Straightline extraction for every linear code

The IORs that we presented are used to obtain a corresponding accumulation scheme ACC in the
ROM, which is a non-interactive protocol. This requires the IORs to satisfy (an appropriate flavor
of) state-restoration security [BCS16; CY24]. Specifically, in order to ensure that ACC satisfies
straightline knowledge soundness in the ROM, the IORs must satisfy straightline state-restoration
knowledge soundness.5
We prove that the IORs in this paper satisfy the aforementioned security notion, even if the
underlying code C does not have an efficient error-tolerant decoder. This is notable in the lineartime prover setting because there are linear-time-encodable codes of practical interest for which
there are no known efficient error-tolerant decoders.
Achieving straightline extraction for every linear code C, however, demands a different approach
compared to prior work. Indeed, prior work on IOPs/IORs achieves straightline extraction via errortolerant decoders (or else, when no error-tolerant decoder is available, prior work only achieves the
weaker notion of rewinding extraction, e.g., as in [BCGGHJ17; GLSTW23; DP24; ZCF24]).
Below we sketch our approach, and then in Section 2.8 we explain how a new notion of roundby-round knowledge soundness aids in carrying out our approach.
5

Alternatively, one could aim for the weaker notion of rewinding knowledge soundness for ACC, which in turn
would require the IORs to satisfy the weaker notion of rewinding state-restoration knowledge soundness. However
rewinding extraction typically comes with downsides, such as worse error bounds and expected-time extraction.

16

Straightline extraction in an SR game. The security property that we target considers
a state-restoration game, which models the (idealized) Fiat–Shamir security of the public-coin
interactive protocol at hand. Informally, an adversary P̃sr makes a number of moves (up to a budget)
that model queries to the random oracle and receives corresponding responses, and ultimately
outputs a final choice of statement and prover messages. Straightline state-restoration knowledge
soundness considers the probability that P̃sr ’s output convinces the interactive protocol’s verifier
and a straightline extractor Esr fails to find a witness for the statement. The extractor Esr is
straightline in that it receives as input the statement, prover messages, and move-response trace
from the state-restoration game (and, crucially, does not receive black-box access to the adversary).
Prior work has adapted these notions and definitions from IOPs to IORs [BMNW25b].
The challenge. Prior work on IOPs/IORs that achieves straightline extraction relies on efficient
error-tolerant decoding for the underlying code. Most notably, the Reed–Solomon code has efficient
error-tolerant decoders (both in the unique-decoding and list-decoding regimes), which are then
used by straightline extractors (see, e.g., [BBHR18; ACFY24; ACFY25]). Typically, the first prover
message in the protocol allegedly encodes a valid witness, and one proves that, except for a small
error probability, if the adversary outputs an accepting transcript in the state-restoration game
then the first prover message is close to the encoding of a valid witness. The straightline extractor
Esr then uses an error-tolerant decoder on that message to find the valid witness. Crucially, an
efficient extractor Esr cannot do so if there is no efficient error-tolerant decoder (which is the case
for a number of codes of interest). We wish to achieve an efficient straightline extractor for every
linear code C, without relying on error-tolerant decoding.
The hope. In the state-restoration game the straightline extractor Esr receives additional information that we may be able to use to avoid error-tolerant decoding, as we now explain.

The condition “∃ y′⋆ ∈ Bδ (y′ ) , (x′ , y′⋆ , w′ ) ∈ RC ” tells us that u ∈ C[(α, µ), (Pb , β, η)] is δ-close

Our key insight is to leverage mutual correlated agreement for this extraction task. Specifically,
due to the mutual correlated agreement property of C, our IOR is such that, with high probability
over the choice of verifier randomness, each fi agrees with a codeword ui ∈ C[(αi , µi ), (Pb , βi , ηi )]
at (exactly) the locations where f agrees with the codeword u ∈ C[(α, µ), (Pb , β, η)].6
This motivates the following extraction strategy. First, Esr finds the set S ⊆ [n] of locations
where f and u agree; note that S contains at least (1 − δ) · n locations. Next, Esr restricts each
fi to S, by replacing symbols in [n] \ S with ⊥. Finally, for each i ∈ [ℓ], Esr erasure corrects
the restriction of fi to S, thereby obtaining the corresponding codeword ui . The extractor Esr
outputs the desired witnesses (w1 , . . . , wℓ ) := (u1 , . . . , uℓ ).
Crucially, every linear code has an efficient erasure correction algorithm (which consists of solving
a linear system induced by the generator matrix); see [GRS12, Exercise 2.16].
(Note that, in both cases, the extractor Esr does not take advantage of the move-response trace
trsr .) Formalizing the above intuition into an analysis requires some care, as we discuss next.
Remark 2.2 (efficiency benefits of erasure correction). Erasure correction is beneficial even for
linear codes with efficient error-tolerant decoding, because erasure correctors can be (and typically
are) faster than error-tolerant decoders. Fixing a distance δ ∈ (0, δ(C)), an error tolerant decoder
returns all codewords at distance δ from the input word, which implies that the decoder time is at
least Ω(|Λ(C, δ)| · n). Instead, an erasure corrector is only required to output the unique erasure
correction of the input word, and thus may run in time O(n). In fact, Reed–Solomon codes can be
erasure corrected in time O(n) · polylog(n). Since the (straightline state-restoration) extractor Esr
uses either the decoder or the corrector as a subroutine, the improvement in asymptotics implies a
more efficient accumulation scheme (by way of a faster security reduction).

Round-by-round extraction for every linear code

Directly proving state-restoration security for an IOP/IOR can be cumbersome, so one typically
proves that the IOP/IOR satisfies a stronger property that is known to imply (the desired flavor of) state-restoration security (see [CY24] for more). Turning to our setting, the common
approach to show that an IOP/IOR satisfies straightline state-restoration knowledge soundness is
to show that the IOP/IOR satisfies round-by-round knowledge soundness [CCHLRR18; CMS19;
CY24; BMNW25b]. Prior definitions, however, appear too strong. We explain this subtle point
and propose a relaxation that we submit as the “right” notion of RBR knowledge soundness for
IOPs/IORs.7 Crucially, we prove the following: (i) our relaxation still implies straightline staterestoration knowledge soundness; and (ii) our IORs satisfy the relaxation.
Warm-up: RBR soundness. We review the notion of RBR soundness [CCHLRR18], generalized
to IORs [BMNW25b]. Informally, an IOR from R to R′ is RBR sound if its soundness can be
modeled as a sequential composition of one-round reductions (each individually sound). This is
expressed via a state function that captures whether a given instance-transcript pair is good for
the prover.
• Before the interaction (the transcript is empty), the instances that are good for the prover are
those close to the language L(R).
• In intermediate rounds of the protocol, the probability (over the verifier’s randomness for that
round) that a state transitions from bad into good (for the prover) is small.
• After the interaction (the transcript is full), the instance-transcript pairs that are good for the
prover are those that cause the verifier to output instances close to the language L(R′ ).
The round-by-round soundness error upper bounds the transition probability in each round.

We outline how protocols typically satisfy RBR knowledge soundness. For simplicity, assume

output is not a valid witness. Outside of the critical round, the transition probability can be upper
bounded, regardless of whether or not the extractor succeeds (indeed, the extractor can simply be

Choice of parameters. Configuring the WARP protocol requires care, due to tradeoffs that arise
in the choice of linear code C and proximity bound δ.
• Which code C? Choosing a linear-time encodable code is necessary to achieve linear-time accumulation. However, the situation is more nuanced when considering concrete efficiency.
Linear-time encodable codes of practical interest typically have some drawbacks: (i) smaller minimum distance than a Reed–Solomon code of the same rate (Reed–Solomon codes are maximum
distance separable); and (ii) worse (known) bounds for mutual correlated agreement compared
to Reed–Solomon codes. This results in an IOR verifier with larger query complexity and,
consequently, an accumulation verifier with larger hash complexity; in turn, this impacts the
overall prover time of the PCD construction obtained from the accumulation scheme (as the
PCD construction proves correctness of a circuit that includes the accumulation verifier).
In light of this, the quasilinear encoding time of Reed–Solomon codes might be (concretely)
small enough that the better query complexity offsets the performance gains in the prover that
come from using a linear-time encodable code.
• Which proximity bound δ? Suppose C has mutual correlated agreement with proximity radius
δPG and error errPG . Our IORs preserve distance for every proximity bound δ ∈ (0, δPG ) (i.e., the
IORs have proximity radius δ ⋆ := δPG ). We generally recommend setting the proximity bound δ
to be as large as possible (using the largest proximity radius δPG known for C), because larger δ
results in fewer queries and thereby a smaller accumulation verifier.
Smaller choices of δ can, however, offer useful tradeoffs. First, known mutual correlated agreement errors worsen as δ increases, which translates into using larger fields to ensure soundness.
Using a smaller δ might then allow saving on field size, improving the efficiency of field arithmetic for both the prover and verifier. Second, when δ ≤ δ(C)/2 (i.e., the proximity radius is at
most the code’s unique decoding radius), the protocols we presented can be simplified to avoid
using out-of-domain samples. This improves the accumulation prover’s efficiency (and round
complexity). Nevertheless, we suspect that the benefits of choosing a smaller proximity radius
δ are not sufficient to offset the increased verifier complexity.
Carefully navigating these tradeoffs requires an implementation (and suitable optimizations). We
leave such efforts to future work.

Preliminaries

We define objects and state results that we use in this paper. We use the following notation.
• The set of powers of two is 2N := {2n : n ∈ N}.
• The preimage set of t ∈ T under a function f : S → T is f −1 (t) := {s ∈ S : f (s) = t}.
• For two functions f, g : [n] → F and a set S ⊆ [n], we write f (S) = g(S) to mean that f (x) = g(x)
for every x ∈ S. Conversely, if f (S) ̸= g(S), then there exists x ∈ S such that f (x) ̸= g(x).
• For two functions f, g : [n] → F, ∆(f, g) is the fractional Hamming distance between f and g
(the fraction of points in which they disagree). For a set S ⊆ F[n] , ∆(f, S) := minh∈S ∆(f, h)
(or if S is empty, ∆(f, S) := 1).

Relations

For the purposes of this work, a relation R consists of quadruples (i, x, y, w) ∈ R, where i is the index,
x the explicit instance, y the implicit instance, and w the witness. For a relation R = {(i, x, y, w)},
let L(R) := {(i, x, y) : ∃w, (i, x, y, w) ∈ R} be the language induced by R.
Multi-instance relations. For ℓ ∈ N, we define the multi-instance relation
Rℓ := {(i, (x1 , . . . , xℓ ), (y1 , . . . , yℓ ), (w1 , . . . , wℓ )) : ∀ i ∈ [ℓ], (i, xi , yi , wi ) ∈ R} .
Distance from a relation. Let R be a relation. For any statement (i, x, y) (not necessarily in
the language), we associate an oracle, denoted πR (i, x, y). Unless stated otherwise, we define this to
be the implicit instance (if y = (y1 , . . . , yℓ ) is a tuple of words, then πR (i, x, y) should be interpreted
as an interleaved word). Let ΠR (i, x) := {πR (i, x, y) : (i, x, y) ∈ L(R)} be the set of oracles in the
relation induced by a given index and explicit instance. We define the distance of a statement
(i, x, y) from the relation R to be
∆R (i, x, y) := ∆(πR (i, x, y), ΠR (i, x)) .

Interactive oracle reductions

Interactive Oracle Reductions (IORs) [BCGGRS19; BMNW25b] are a generalization of Interactive
Oracle Proofs (IOPs) [BCS16; RRR16]. Below we describe the flavor of IOR relevant to this paper.
A public-coin holographic IOR of proximity from a relation R to R′ is a triple IOR = (I, P, V)
that works as follows.
• Offline phase. The indexer I is a deterministic algorithm that receives as input an index i and
outputs a tuple (i′ , ι, I) consisting of a new index i′ , a short index ι (an explicit input for V),
and a long index I (an implicit input for V).
In this paper, all IORs satisfy i′ = i (the index is unchanged) and I = ⊥ (no long index). To
simplify notation, we write I(i) = ι (I is tasked to output the short index ι only).
• Online phase. The prover P is an interactive algorithm that receives as input (i, x, y, w) ∈ R,
and the verifier V is an interactive oracle algorithm that receives as (explicit) input (ι, x) and
query access to y. They interact over k rounds as follows. In each round i ∈ [k], P sends a proof
string πi and V sends a uniformly random message ρi ∈ {0, 1}ri . After k rounds of interaction,
V makes queries to y and proof strings π1 , . . . , πk sent by P. After that:
– V either rejects or outputs a new explicit instance x′ and implicit instance y′ , and
– P outputs a new witness w′ such that (i, x′ , y′ , w′ ) ∈ R′ .
Without loss of generality, V’s output is a deterministic function of its input and the transcript.
Note that V specifies y′ implicitly, in terms of y and π1 , . . . , πk .

Codes

We state several definitions and facts about (error-correcting) codes.
Definition 3.6. A code over an alphabet Σ with message length k and codeword length n is an
injective map C : Σk → Σn . A code C is a linear code if Σ = F is a field and C is linear. In this
paper we assume that n ∈ 2N .
We often view C ⊆ Σn as the map’s image and codewords u ∈ C as functions u : [n] → Σ.
Definition 3.7. The minimum distance of a code C is defined as δ(C) := minu̸=v∈C ∆(u, v).
Definition 3.8. A code C : Σk → Σn is systematic if, for all x ∈ Σk , the first k entries of C(x)
are equal to x.
Algorithmic measures. We consider several algorithmic measures associated to C.
• encC is the number of field operations required to encode a message x ∈ Σk into its corresponding
codeword u := C(x).
• decC is the number of field operations required to decode a codeword u ∈ C into its corresponding
message x := C −1 (u); if C is systematic, then decC = 0 (since x is the first k elements of u).
• ecorC is the number of field operations required to erasure correct a codeword (Definition 3.9);
every linear code has efficient erasure correction (Lemma 3.10).
Definition 3.9. A code C ⊆ Σn supports erasure correction with correction time ecorC if
there exists a deterministic algorithm EC (the erasure corrector) that, given input f : [n] → Σ∪{⊥}:
• if |f −1 (⊥)| < δ(C) · n and there exists a codeword u ∈ C such that f (i) = u(i) for all i ∈
[n] \ f −1 (⊥), then EC (f ) = u (note that such a codeword u is unique); and
• otherwise, EC (f ) = ⊥.
Moreover, EC performs at most ecorC field operations.
Lemma 3.10 ([GRS12]). Every linear code C : Fk → Fn has erasure correction with correction time
ecorC = O(n3 ) .

define the list decoding of f within distance δ as
Λ(C, f, δ) := {u ∈ C | ∆(f, u) ≤ δ} .

Succinct arguments vs. accumulation schemes

A key efficiency metric of PCD is the recursion overhead: the additional computation whose correctness is proven beyond the PCD predicate’s correctness. The recursion overhead differs across
PCD constructions. Below ℓ refers to the arity of the PCD predicate.
1. For PCD based on a succinct non-interactive argument ARG = (PARG , VARG ), the recursion
overhead is ℓ executions of VARG .
2. For PCD based on an accumulation scheme ACC = (PACC , VACC , DACC ), the recursion overhead is
1 execution of VACC on ℓ instance-witness pairs and ℓ accumulators.
The latter is typically more efficient, and motivates the study of accumulation schemes. A detailed
quantitative comparison, however, is more nuanced due to the fact that a succinct non-interactive
argument leads to PCD via an alternate “indirect” construction that yields different tradeoffs.
A succinct non-interactive argument ARG = (PARG , VARG ) implies a corresponding accumulation
scheme ACC = (PACC , VACC , DACC ) with a trivial decider DACC : (i) PACC runs PARG on the (conjunction
of) ℓ instance-witness pairs; (ii) VACC runs VARG on the ℓ instances; and (iii) DACC always accepts.
One can then construct PCD from this accumulation scheme ACC (following Item 2 above).14
In particular, if PARG is linear time then so is PACC , which directly yields constructions of
linear-time accumulation schemes from corresponding linear-time succinct non-interactive arguments [BCGGHJ17; BCG20; BCL22; XZS22; RR22; GLSTW23; BCFRRZ25; NST24; BMMS25].
The corresponding accumulation schemes are incomparable to WARP in the following sense. (Below
λ is the security parameter and, for concreteness, we consider the PESAT relation from Definition 1.)
• The verifier VACC of accumulation schemes obtained via direct “non-succinct” constructions
typically runs in time ℓ · f (λ, k) for some function f (think f = Oλ (log k)).
• The verifier VACC of accumulation schemes obtained from succinct arguments typically run in
time g(λ, ℓ · (M + N )) for some function g = ω(f ).
These costs are incomparable in that the first approach is better for small to medium arities ℓ (e.g.,
IVC corresponds to ℓ = 1), whereas the second approach is better for larger arities ℓ. In this paper
we focus on direct accumulation schemes (and the regime where they are better).

14

This indirect construction is not the same as the direct PCD construction: recursive proof composition of ARG
yields succinct PCD proofs whereas PCD based on ACC does not. On the other hand, using ACC leads to a smaller
recursion overhead, as we informally explain next. Let VARG (|C|) denote the complexity of VARG on a circuit of size |C|.
In the direct PCD construction, the recursion overhead is ℓ·VARG (|C|). In the indirect PCD construction the recursion
overhead is VARG (ℓ · |C|). Hence the indirect construction may be beneficial to recursion overhead: VARG (ℓ · |C|) can
be smaller than ℓ · VARG (|C|) depending on the arity ℓ.

We prove that our variant of round-by-round knowledge soundness (Section 4) is a relaxation of
prior definitions [CMS19; BMNW25b], in the sense that any IOR that is knowledge sound under

Extension fields

Throughout the paper we considered a PESAT relation R⊚ (F) and a linear code C : Fk → Fn defined
over a field F that is required to be large enough for soundness. Here we discuss how to adapt
our two main IORs to the setting of field extensions: the PESAT relation R⊚ (B) and linear code
CB : Bk → Bn are defined over a base field B (e.g., B = F2 ), and we wish to benefit from a larger
field extension F (e.g., F = F2160 ) for soundness. We wish to adjust our IORs so that the verifier
samples field elements from F (rather than B) while using B for the reduction from PESAT.

D.1

Definitions

We review definitions about extension fields and codes defined over extension fields.
Definition D.1. A tuple (B, F, e, ψ, φ) is a extension field presentation if:
• B, F are fields;
• ψ : B ,→ F is an injective field homomorphism;
• e ∈ N is the dimension of F as a B-vector space;
• φ : F → Be is a B-linear isomorphism.
For each i ∈ [e], φi : F → B denotes the i-th component of φ. An extension field presentation
(B, F, e, ψ, φ) is systematic if, for every x ∈ B, φ(ψ(x)) = (x, 0, . . . , 0).
We also apply ψ to vectors of base field elements and φ, φ1 , . . . , φe to vectors of extension field
elements by applying each of these maps componentwise.
An extension code defines a code over a field extension for a linear code over the base field.
Definition D.2. Let CB : Bk → Bn be a linear code. The extension code of CB over F (with
respect to the extension field presentation (B, F, e, ψ, φ)) is the linear code CF : Fk → Fn defined as

CF (v) := φ−1 CB (φ1 (v)), . . . , CB (φe (v)) .

[DP25, Thm 3.2].
The list size of the extension code is the list size of the e-interleaved base code (which in turn
is related to the list size of the base code by Lemma 3.13).

D.3

Adapting the accumulation scheme

Theorem 10.3 and Construction 10.4 can be adapted to handle base and extension fields by compiling the adapted IORs described in Appendix D.2. The soundness analysis is identical, and follows
the sketch in Section 10.2. Concretely, this results in a verifier for ACC similar to that in the (unmodified) WARP construction, which additionally checks that opened locations are in the relevant
alphabet of the Merkle commitment (base field elements in B or extension field elements in F).
Remark D.4. Construction 10.4 benefits particularly from IORWARP running Construction 5.10 in
parallel (rather than sequentially). Sequentially, committing to ℓ1 encoded witnesses over the base
field would involve O(ℓ1 · n) random oracle queries of size log |B|. By running Construction 5.10 in
parallel instead, this cost is reduced to O(n) random oracle queries of size ℓ1 · log |B|. If ℓ1 ≈ e, then
this corresponds to the cost of committing to a single encoded witness over the extension field.
